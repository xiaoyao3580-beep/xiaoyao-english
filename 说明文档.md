
## 1. 这个系统是做什么的

这个项目是一个给学生和老师使用的线上学习中心。

它主要解决几件事：

1. 学生登录后，可以看到自己有权限进入的课程。
2. 学生可以打开课程作业、练习页面、投票页面等内容。
3. 老师登录后，可以管理学生、课程作业、首页宣传图、打卡、成绩报表、投票结果和宠物成长系统。
4. 系统会把学生练习、打卡、成绩、宠物奖励等数据存在 Supabase 里，方便后续查看和统计。

一句话概括：

> 这是一个“课程入口 + 老师后台 + 学生数据记录 + 趣味成长系统”的网页应用。

---

## 2. 当前实现目标

系统当前的目标不是做成一个特别复杂的大平台，而是做成一个稳定、好维护、老师能用、学生能打开的教学网站。

核心目标如下：

| 目标 | 解释 |
| --- | --- |
| 稳定打开 | 学生点开网页不能白屏，能快速进入课程 |
| 登录分权限 | 不同学生只能看到自己班级的内容，老师能进入后台 |
| 课程集中管理 | 所有课程和作业入口集中在首页和后台 |
| 数据可保存 | 学生练习记录、成绩、打卡、投票都能保存 |
| 老师易维护 | 老师后台可以直接增删学生、发布作业、换首页图 |
| 有成长反馈 | 宠物、积分、经验值给学生一点游戏化反馈 |
| 少依赖复杂框架 | 主系统用原生 JavaScript，减少“依赖坏了就白屏”的风险 |

## 3. 技术栈总览

这里的“技术栈”可以理解为：

> 这个网站用了哪些工具、语言和服务来跑起来。

| 技术 | 在项目里做什么 | 大白话解释 |

## 5. 系统整体架构

可以把系统想成三层：

```text
学生 / 老师
   ↓
浏览器网页
   ↓
Supabase 云端数据库和文件仓库
```

### 5.1 第一层：用户

用户分两类：

| 用户 | 能做什么 |
| --- | --- |
| 学生 | 登录、看课程、做作业、提交练习、参与投票、查看宠物成长 |
| 老师 | 管学生、管作业、管首页图、看成绩、管打卡、管投票、管宠物规则 |

### 5.2 第二层：浏览器网页

浏览器加载这些文件：

```text
index.html
  ↓
assets/tailwind.css
  ↓
assets/vendor/supabase/supabase.js
  ↓
content-data.js
  ↓
vote-data.js
  ↓
app.js
```

中文注释版说明：

```text
index.html                  # 先打开网页壳子
assets/tailwind.css         # 加载页面样式
supabase.js                 # 准备连接云端数据库
content-data.js             # 读取课程目录
vote-data.js                # 读取投票题目
app.js                      # 启动整个系统
```

### 5.3 第三层：Supabase 云端

Supabase 在这里主要做两件事：

1. 数据库存文字数据，比如学生、作业、成绩、投票。
2. Storage 存文件，比如课程 HTML、banner 图片、宠物图片。

## 6. 前端页面是怎么工作的

主页面不是很多个独立页面跳来跳去，而是主要由 `app.js` 控制。

它会根据当前状态决定显示什么：

| 状态 | 页面 |
| --- | --- |
| `home` | 首页课程列表 |
| `level` | 某个课程等级的作业列表 |
| `teacher` | 老师后台 |
| `vote` | 投票页面 |

`app.js` 里有一个 `state` 对象，可以理解成“当前页面记事本”。

例如：

```javascript
// 中文注释：state 记录当前页面是什么、当前课程是什么、数据有没有加载完
const state = {
  page: 'home',
  level: null,
  students: [],
  homework: [],
  banners: [],
  loading: true
};
```

当用户点击按钮时，系统会更新 `state`，然后重新渲染页面。

大白话：

> 用户点哪里，`app.js` 就改一下“当前状态”，再把页面重新画出来。

## 7. 登录和权限逻辑

### 7.1 老师登录

老师账号写在 `app.js` 里：

```javascript
// 中文注释：这是老师后台账号
const TEACHER = {
  id: 'xiaoyao',
  password: '929292',
  name: '肖瑶老师'
};
```

老师登录成功后，会进入老师后台。

### 7.2 学生登录

学生账号存在 Supabase 的 `students` 表里。

登录时，系统会去 Supabase 查：

```text
账号是否存在？
密码是否正确？
这个学生属于哪些班级？
```

如果正确，就把登录信息存在浏览器的 `localStorage` 里。

大白话：

> `localStorage` 就是浏览器的小本子，用来临时记住“现在登录的是谁”。

### 7.3 课程权限

学生账号里有一个 `classes` 字段，表示这个学生能进入哪些课程。

例如：

```text
S01 → 只能看 A2
S02 → 可以看 A2 和 F2
老师 → 所有课程都能看
```

课程页面里还有 `lesson-auth-guard.js`，它会再次检查权限。

这样做的目的：

> 防止学生绕过首页，直接复制课程链接打开不属于自己的课程。

## 8. 课程和作业体系

课程目录主要在 `content-data.js` 里。

它有两类数据：

| 数据 | 作用 |
| --- | --- |
| `XY_CONTENT_LEVELS` | 有哪些课程大类，比如 A1、A2、F2 |
| `XY_CONTENT_MODULES` | 每个课程大类下面有哪些作业或模块 |

一个课程模块大概长这样：

```javascript
{
  id: 'a2-unit2-review',                    // 中文注释：作业唯一编号
  unitCode: 'Unit 2',                       // 中文注释：单元名
  title: 'Unit 2 综合复习',                  // 中文注释：页面显示标题
  description: '课文精讲 / 词汇打卡 / 口语大师', // 中文注释：简单介绍
  localPath: 'classes/lessons/a2-unit2-review.html' // 中文注释：真实课程文件位置
}
```

作业文件大多放在：

```text
classes/lessons/
```

老师后台也可以把新的课程 HTML 上传到 Supabase Storage 的 `lesson-files` 桶里。

## 9. 老师后台功能

老师后台由 `app.js` 里的 `teacherPage()` 和相关函数生成。

目前主要分成这些模块：

| 后台模块 | 能做什么 |
| --- | --- |
| 学生管理 | 新增学生、改密码、改姓名、改班级权限、删除学生 |

| 投票管理 | 查看投票提交，导出 CSV，删除提交 |
| 宠物系统 | 创建宠物、调整经验和积分、配置道具和奖励规则 |

## 11. 宠物成长系统

宠物系统的作用是让学生完成练习后获得反馈。

它大概这样工作：

```text
学生完成练习
   ↓
课程页面写入 growth_logs
   ↓
Supabase 触发宠物奖励逻辑
   ↓
学生宠物获得经验值和积分
   ↓
学生可以买道具、改名、换背景、解锁新宠物
```

用到的核心表：

| 表 | 作用 |
| --- | --- |
| `student_pets` | 当前宠物是什么、叫什么、多少经验、多少积分 |
| `pet_inventory` | 已买了哪些道具 |
| `pet_reward_events` | 每次加分、扣分、奖励的记录 |
| `pet_reward_rules` | 分数达到多少，奖励多少经验和积分 |

老师后台可以手动调整宠物经验、积分和解锁情况。

## 12. 投票系统

投票题目写在 `vote-data.js` 里。

它包括：

| 数据 | 作用 |
| --- | --- |
| `classOptions` | 可以选择的班级 |
| `templates` | 不同类型题目的答题结构 |
| `topics` | 具体投票题目 |

学生提交投票时，会调用 Supabase 的函数：

```text
submit_vote_selection
```

这个函数负责检查规则并写入 `vote_submissions` 表。

老师后台可以查看投票结果，也可以导出 CSV。

## 13. 成绩和打卡系统

### 13.1 成绩来源

课程页面或练习页面会把学生的完成结果写到：

```text
growth_logs
```

里面通常会有：

| 字段内容 | 大白话解释 |
| --- | --- |
| 学生 ID | 是谁做的 |
| 作业 ID | 做的是哪个作业 |
| 分数 | 做得怎么样 |
| 正确数 / 总数 | 答对多少题 |
| 提交时间 | 什么时候做的 |

老师后台的成绩报表会从这些记录里统计平均正确率、提交次数、趋势和排名。

### 13.2 打卡和豁免

打卡相关数据在：

```text
attendance_records
```

它主要用于老师手动标记某个时间段学生是否豁免或已处理。

## 16. 平时做网页要注意的地方

这一节是日常最容易踩坑的地方，尤其是新写一个“作业网页”或“练习网页”时。

简单说：

> 作业网页不只是能打开、能做题，还要能登录校验、能记录成绩、能被老师后台统计。

### 16.1 作业网页必须有清楚的作业 ID

每个作业页面都要能对应到一个唯一 ID。

这个 ID 通常来自 `content-data.js` 里的模块 `id`。

例如：

```javascript
{
  id: 'a2-unit2-review', // 中文注释：这个就是作业 ID
  title: 'Unit 2 综合复习',
  localPath: 'classes/lessons/a2-unit2-review.html'
}
```

作业页面里最好也写上：

```html
<!-- 中文注释：告诉权限检查和成绩记录，这个页面属于哪个作业 -->
<body data-module-id="a2-unit2-review">
</body>
```

注意：

> `content-data.js` 里的 `id`、页面里的 `data-module-id`、成绩记录里的 `homework_id`，最好保持完全一样。

### 16.2 作业页要加权限保护

放在 `classes/lessons/` 里的作业入口页，推荐加：

```html
<!-- 中文注释：检查学生有没有权限打开这个作业 -->
<script src="./lesson-auth-guard.js?v=20260512-auth-guard"></script>
```

如果页面路径不同，要注意相对路径是否正确。

常见写法：

```html
<!doctype html>
<html lang="zh-CN">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1, viewport-fit=cover">
  <title>Unit 2 综合复习 | Xiaoyao Studio</title>
</head>
<body data-module-id="a2-unit2-review">
  <script src="./lesson-auth-guard.js?v=20260512-auth-guard"></script>
  <iframe
    data-src="../_yu1/lesson-standalone.html?v=20260511-lesson-only#/level/a2/A2Unit2Review"
    allow="autoplay; clipboard-read; clipboard-write; fullscreen; microphone; camera"
    allowfullscreen>
  </iframe>
</body>
</html>
```

大白话：

> 不加权限保护，学生可能直接复制链接打开不属于自己的课程。

### 16.3 作业结尾必须能记录打卡和成绩

学生完成作业后，系统要往 Supabase 的 `growth_logs` 表写一条记录。

这条记录会影响：

1. 老师后台成绩统计。
2. 打卡完成情况。
3. 宠物经验和积分奖励。

最关键字段：

| 字段 | 大白话解释 |
| --- | --- |
| `student_id` | 哪个学生 |
| `homework_id` | 哪份作业 |
| `score` | 得分或正确率，0 到 100 |
| `correct_count` | 答对几题 |
| `total_count` | 总共几题 |
| `module_title` | 作业标题 |
| `event_type` | 记录类型，比如 `score_complete` |

### 16.4 最推荐的成绩记录方式

如果你自己写作业网页，最稳的方式是在结束时主动调用：

```javascript
// 中文注释：学生完成作业后，主动告诉系统成绩
window.XY_RECORD_SCORE({
  homework_id: 'a2-unit2-review',
  module_title: 'Unit 2 综合复习',
  score: 86,
  correct_count: 43,
  total_count: 50,
  event_type: 'score_complete'
});
```

注意：

1. `score` 推荐传 0 到 100 的整数。
2. 有题目数量时，最好同时传 `correct_count` 和 `total_count`。
3. `homework_id` 必须和 `content-data.js` 里的作业 `id` 对上。
4. 只有学生登录时才会记录；老师预览通常不会记入学生成绩。

### 16.5 自动记录成绩的方式

项目里有一个记录桥接文件：

```text
classes/_yu1/xy-record-bridge.js
```

它会自动观察页面：

1. 有没有点击“提交、完成、保存、生成报告、查看结果、交卷、打卡”等按钮。
2. 页面结尾有没有出现“正确率、答对题数、Final Score、Accuracy、成绩报告”等结果文字。
3. 有没有出现类似 `43 / 50` 或 `86%` 的成绩。

如果它识别到结果，就会写入 `growth_logs`。

为了让自动识别更稳定，作业结尾建议显示这样的内容：

```html
<!-- 中文注释：这些关键词能帮助系统识别最终成绩 -->
<section id="final-report">
  <h2>成绩报告</h2>
  <p>答对题数：43 / 50</p>
  <p>正确率：86%</p>
  <button>完成打卡</button>
</section>
```

不推荐只写：

```html
<p>太棒了！</p>
```

因为系统看不出具体成绩。

### 16.6 什么时候需要手动引入 `xy-record-bridge.js`

如果课程是通过 `classes/lessons/player.html` 打开的，系统会自动注入 `xy-record-bridge.js`。

如果你写的是一个独立 HTML 作业页，建议手动加：

```html
<!-- 中文注释：负责把完成记录和成绩写入 growth_logs -->
<script src="../_yu1/xy-record-bridge.js?v=20260511-content-sync" defer></script>
```

路径要根据文件位置调整。

例如：

| 当前页面位置 | 推荐路径 |
| --- | --- |
| `classes/lessons/xxx.html` | `../_yu1/xy-record-bridge.js` |
| `classes/_yu1/xxx.html` | `./xy-record-bridge.js` |

### 16.7 作业网页的库使用要求

作业网页可以用库，但要尽量用项目本地已有的库。

原因很简单：

> 外部 CDN 有时会慢、会打不开；本地库更稳定。

项目本地已有这些常用库：

| 库 | 本地位置 | 用来做什么 |
| --- | --- | --- |
| React | `assets/vendor/react/react-18.production.min.js` | 写复杂互动页面 |
| ReactDOM | `assets/vendor/react/react-dom-18.production.min.js` | 把 React 页面显示出来 |
| Babel | `assets/vendor/babel/babel-standalone.min.js` | 让浏览器直接跑 JSX |
| Tailwind 运行时 | `assets/vendor/tailwind/tailwindcss-3.4.17.js` | 临时支持 Tailwind 类名 |
| 编译后 Tailwind CSS | `assets/tailwind.css` | 主站样式 |
| Font Awesome | `assets/vendor/fontawesome-6.5.2/css/all.min.css` | 图标 |
| html2canvas | `assets/vendor/html2canvas/html2canvas-1.4.1.min.js` | 把页面截图成图片 |
| Supabase UMD | `assets/vendor/supabase/supabase.js` | 连接 Supabase |

### 16.8 不推荐直接写外部 CDN

不推荐在新作业页里直接写：

```html
<script src="https://cdn.tailwindcss.com"></script>
<script src="https://unpkg.com/react@18/umd/react.production.min.js"></script>
<script src="https://unpkg.com/@babel/standalone/babel.min.js"></script>
<link rel="stylesheet" href="https://cdnjs.cloudflare.com/ajax/libs/font-awesome/6.4.0/css/all.min.css">
```

更推荐用本地资源，或者让 `player.html` 自动替换成本地资源。

大白话：

> 新页面能不用外链就不用外链。外链一挂，学生那边就可能白屏。

### 16.9 新作业页最稳的资源写法

如果是放在 `classes/lessons/` 下的独立作业页，可以这样写：

```html
<!-- 中文注释：本地 Tailwind 样式 -->
<link rel="stylesheet" href="../../assets/tailwind.css">

<!-- 中文注释：本地图标库 -->
<link rel="stylesheet" href="../../assets/vendor/fontawesome-6.5.2/css/all.min.css">

<!-- 中文注释：如果需要 React，再引入本地 React -->
<script src="../../assets/vendor/react/react-18.production.min.js"></script>
<script src="../../assets/vendor/react/react-dom-18.production.min.js"></script>

<!-- 中文注释：如果写 JSX，才需要 Babel；不用 JSX 就别加 -->
<script src="../../assets/vendor/babel/babel-standalone.min.js"></script>
```

注意：

> Babel 会让页面更重。能用普通 HTML、CSS、JS 完成的作业页，就不要硬上 React + Babel。

### 16.10 作业网页结尾推荐结构

每个有成绩的作业，结尾最好统一有：

1. 明确的完成按钮。
2. 明确的成绩文字。
3. 明确的正确题数。
4. 明确的打卡成功提示。
5. 主动调用 `window.XY_RECORD_SCORE(...)`。

推荐示例：

```html
<section id="result">
  <h2>成绩报告</h2>
  <p id="scoreText">正确率：86%</p>
  <p id="countText">答对题数：43 / 50</p>
  <button id="finishBtn">完成打卡</button>
</section>

<script>
  // 中文注释：点击完成按钮时，把成绩写进系统
  document.getElementById('finishBtn').addEventListener('click', function () {
    window.XY_RECORD_SCORE({
      homework_id: 'a2-unit2-review',
      module_title: 'Unit 2 综合复习',
      score: 86,
      correct_count: 43,
      total_count: 50,
      event_type: 'score_complete'
    });
  });
</script>
```

### 16.11 作业网页不要做的事

尽量不要这样做：

1. 不写 `data-module-id`。
2. 作业 ID 和 `content-data.js` 对不上。
3. 页面最后没有成绩，只写“完成了”。
4. 只在页面上显示成绩，但不调用记录函数。
5. 大量使用外部 CDN。
6. 把 Supabase 的 service role key 写进网页。
7. 把学生隐私信息写死在 HTML 里。
8. 只在自己电脑测试，不用学生账号测试。

### 16.12 新作业页上线前检查清单

上线前建议逐项确认：

```text
[ ] 页面能正常打开
[ ] 手机屏幕不溢出、不挡按钮
[ ] 已加 data-module-id
[ ] 已加 lesson-auth-guard.js
[ ] 作业 ID 和 content-data.js 一致
[ ] 结尾有“成绩报告 / 正确率 / 答对题数”
[ ] 点击完成后能写入 growth_logs
[ ] 老师后台成绩报表能看到记录
[ ] 宠物经验或积分能正常变化
[ ] 学生无权限时不能直接打开
[ ] 没有依赖打不开的外部 CDN
```

## 17. 常见维护任务

### 17.1 新增一个课程入口

主要改：

```text
content-data.js
```

需要做：

1. 在对应课程大类的 `XY_CONTENT_MODULES` 里加一条模块数据。
2. 把课程 HTML 放到 `classes/lessons/`，或者通过老师后台上传到 `lesson-files`。
3. 确认学生账号的 `classes` 里包含对应课程大类。

### 17.2 新增学生

推荐从老师后台新增。

后台会写入 Supabase 的：

```text
students
```

如果直接改数据库，要保证至少有：

```text
id         # 学生账号
password   # 登录密码
name       # 学生姓名
classes    # 学生可进入的班级列表
```

### 17.3 修改学生课程权限

改学生的 `classes` 字段。

大白话：

> 学生能不能进某个课程，主要看他的 `classes` 里有没有那个课程 ID。

常见课程 ID：

| ID | 课程 |
| --- | --- |
| `f2` | TALENT F2 |
| `a1` | TALENT A1 |
| `a1-plus` | TALENT A1+ |
| `a2` | TALENT A2 |
| `a2-plus` | TALENT A2+ |
| `swsy` | 深外素养口语课 |
| `junior-ability` | 初中能力提升 |
| `economist` | 经济学人精品课 |
| `others` | 成人零基础口语 |

### 17.4 修改首页宣传图

推荐从老师后台的 banner 管理里改。

对应数据表是：

```text
banners
```

图片一般存在：

```text
banner-images
```

### 17.5 更新 Tailwind 样式

如果修改了 `src/tailwind.css` 或新增了很多 Tailwind 类名，运行：

```bash
npm run build:css
```

开发时也可以运行：

```bash
npm run watch:css
```

### 17.6 增加新的 Supabase 表或字段

建议做法：

1. 新建一个清楚命名的 SQL 文件。
2. 写 `create table if not exists` 或 `alter table add column if not exists`。
3. 在 Supabase SQL Editor 里执行。
4. 再改 `app.js` 读取或写入新字段。

这样做的好处：

> 以后别人看到 SQL 文件，就知道数据库是怎么一步步建出来的。

## 20. 安全说明

这个项目是前端直连 Supabase。

这意味着：

```text
浏览器网页
  ↓
直接读写 Supabase
```

这种方式简单、开发快、适合小型教学系统，但也有一些限制：

| 点 | 说明 |
| --- | --- |
| Supabase publishable key 在前端 | 这是公开 key，不等于管理员密钥，但仍要配合数据库规则使用 |
| 老师密码写在前端 | 简单好用，但安全性不适合大型商业系统 |
| 多个表的策略比较开放 | 方便前端操作，但要避免存特别敏感的数据 |
| 学生密码是明文逻辑 | 适合内部教学账号，不适合正式互联网账号系统 |

建议：

1. 不要在里面存身份证、家庭住址、银行卡等敏感信息。
2. 老师密码要定期换。
3. 如果以后用户量变大，应考虑改成真正的后端登录系统。
4. Supabase 的 service role key 绝对不要放进前端代码。

## 21. 常见故障排查

### 21.1 页面空白

先检查：

1. `index.html` 是否正常加载。
2. 浏览器控制台有没有 JS 报错。
3. `app.js` 是否 404。
4. `content-data.js` 和 `vote-data.js` 是否 404。
5. `assets/vendor/supabase/supabase.js` 是否能加载。

### 21.2 学生登录失败

检查：

1. `students` 表里有没有这个账号。
2. 密码是否一致。
3. 学生 ID 是否大小写不一致。
4. Supabase 是否连接失败。

### 21.3 学生看不到课程

检查：

1. 学生的 `classes` 是否包含课程 ID。
2. `content-data.js` 里课程 ID 是否写对。
3. 课程模块的 `localPath` 是否存在。
4. 是否被 `lesson-auth-guard.js` 拦截。

### 21.4 首页 banner 不显示

检查：

1. `banners` 表里有没有数据。
2. `status` 是否是 `open`。
3. `image` 图片地址是否能打开。
4. 图片是否存在于 `banner-images` 桶。

### 21.5 宠物不加经验

检查：

1. 课程是否成功写入 `growth_logs`。
2. `supabase-pet-setup.sql` 是否执行过。
3. `pet_reward_rules` 是否有奖励规则。
4. Supabase 触发器是否正常。

### 21.6 样式不对

检查：

1. `assets/tailwind.css` 是否存在。
2. 修改 Tailwind 后有没有运行 `npm run build:css`。
3. 浏览器是否缓存了旧 CSS。

## 22. 修改代码时的建议

### 22.1 优先改小地方

不要一次性大改很多文件。

推荐顺序：

```text
先改数据
再改页面
最后改数据库
```

### 22.2 改之前先备份

尤其是这些文件：

```text
app.js
content-data.js
vote-data.js
supabase-*.sql
```

### 22.3 保持命名统一

课程 ID、作业 ID、学生班级权限要统一。

例如：

```text
a2-plus
```

不要一会儿写：

```text
A2+
a2plus
a2_plus
```

否则权限和课程匹配会出问题。

### 22.4 修改数据库后要同步前端

如果数据库字段变了，`app.js` 里读取字段的地方也要跟着改。

大白话：

> 数据库换了抽屉名字，前端也要知道新抽屉叫什么。

## 23. 快速理解版

如果只想快速理解这个项目，记住下面这几句话就够了：

1. `index.html` 是入口。
2. `app.js` 是主程序，控制页面和后台。
3. `content-data.js` 管课程目录。
4. `vote-data.js` 管投票题目。
5. `classes/` 放具体课程页面。
6. Supabase 存学生、作业、成绩、打卡、投票、宠物数据。
7. `supabase-*.sql` 是创建数据库表的说明书。
8. `assets/` 放样式、图标和第三方库。
9. 改样式后可能要运行 `npm run build:css`。
10. 部署时不要漏掉 `assets/`、`classes/` 和几个核心 JS 文件。
